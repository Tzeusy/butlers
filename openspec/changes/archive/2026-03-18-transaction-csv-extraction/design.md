## Context

The finance butler ingests transactions one-at-a-time from email notifications via `record_transaction` / `record_transaction_fact`. Users with months or years of bank/credit card CSV exports have no import path. CSV formats vary wildly across institutions (different column names, date formats, amount conventions) and change over time from the same source, so rigid column-mapping is brittle.

The butler's runtime is an ephemeral LLM CLI session (OpenCode adapter) that can read files, write scripts, and execute them — in addition to calling MCP tools. This makes a script-generation workflow natural.

Current deduplication relies on `source_message_id` (email provenance). CSV rows have no such identifier, requiring a composite key.

## Goals / Non-Goals

**Goals:**
- Enable bulk import of CSV transaction history through an adaptive, LLM-driven workflow
- Keep CSV data out of the LLM's context window (token budget protection)
- Make re-imports idempotent via composite deduplication
- Provide both an HTTP endpoint (for generated scripts) and an MCP tool (for direct butler use)

**Non-Goals:**
- OFX/QFX/Plaid integration (separate feature)
- Merchant name normalization (separate concern — "WHOLEFDS MKT #10456" → "Whole Foods")
- Balance tracking or reconciliation
- Statement PDF parsing
- Interactive human-in-the-loop validation of column mappings

## Decisions

### 1. Script-generation workflow (not full-file context, not per-row tool calls)

The LLM samples the CSV (header + first ~10 rows), writes a disposable Python script that parses the full file and POSTs normalized batches to a bulk HTTP endpoint, then executes the script.

**Why:** Keeps the full CSV out of the LLM context (a 5,000-row CSV would consume ~500K tokens — ruinous). The LLM adapts to whatever format it sees without rigid parsing rules. Self-corrects on script errors naturally (read stderr, fix, re-run).

**Alternative rejected — load full CSV into context:** Token budget catastrophe on any real statement file.

**Alternative rejected — per-row `record_transaction` calls:** 500 tool calls for a small statement; the MCP round-trip overhead alone would be minutes, and token cost for 500 tool-call/response cycles dwarfs a single script execution.

**Alternative rejected — profile-per-account mapping:** Formats change over time from the same institution; profiles become stale maintenance burden.

### 2. Bulk endpoint on dashboard API

`POST /api/finance/transactions/bulk` accepts an array of normalized transaction objects, persists via the existing fact layer, and returns per-row results (imported/skipped/error).

**Why:** HTTP is naturally callable from generated Python scripts (`requests.post` or `urllib`). The dashboard API already has connection pool wiring and the finance schema. Scripts don't need MCP access.

**Also:** A `bulk_record_transactions` MCP tool wraps the same logic for direct butler use (e.g., batching multiple email-extracted records in a single call). But the skill's primary path is HTTP POST from a generated script.

### 3. Composite dedup for CSV rows

CSV rows have no `source_message_id`. Dedup uses `sha256(posted_at|amount|merchant|account_id)` as the idempotency key passed to `store_fact`. This is the same `idempotency_key` mechanism already used by `record_transaction_fact`.

**Why:** Makes re-importing the same CSV safe. Different transactions at the same merchant on the same day with different amounts are distinct.

**Risk:** Genuine duplicate transactions (exact same merchant, amount, date, account) would be deduped. **Mitigation:** The import result includes a `skipped` count. The skill instructs the LLM to warn the user when skipped count is high relative to total, suggesting they verify whether duplicates were intentional (e.g., two separate $5.00 coffees on the same day).

### 4. Skill prompting as the control plane

The `transaction-csv-extraction` skill prompt is the **only** mechanism controlling the butler's runtime behavior during CSV import. The prompt MUST:

- **Explicitly instruct** the write-script-then-POST workflow
- **Explicitly forbid** reading the entire CSV into context or calling `record_transaction` per row
- **State the reason** (token budget) so the LLM understands the constraint and doesn't "optimize" it away
- **Provide the exact endpoint URL pattern** and request/response schema so the LLM generates correct scripts on the first try

Without emphatic, unambiguous prompting, the LLM will default to the simpler (but catastrophic) pattern of reading the full file and calling single-record tools in a loop.

### 5. Generated script contract

The disposable Python script generated by the LLM MUST:

- Use stdlib `csv` module (no `pandas` or other dependencies)
- Use stdlib `urllib.request` or `json` for HTTP (no `requests` dependency assumed)
- POST in batches of 100 records to avoid payload size issues
- Print a JSON summary to stdout: `{"total": N, "imported": N, "skipped": N, "errors": N}`
- Exit with code 0 on success, non-zero on fatal errors
- Handle common CSV edge cases: BOM markers, quoted fields, empty trailing rows

**Why stdlib-only:** The butler's runtime environment may not have `requests` or `pandas` installed. stdlib `csv` + `urllib.request` are always available.

### 6. Skip embeddings for bulk-ingested transactions

`store_fact()` computes one SentenceTransformer embedding per call (~50-100ms on CPU). For 500 rows, that's 25-50 seconds — unacceptable for a synchronous HTTP request.

Transaction facts are structured data queryable via metadata JSONB fields (merchant, amount, category, date). Semantic vector search is not a useful retrieval path — no one searches "that grocery purchase last week" by embedding similarity. The tsvector full-text search path remains useful and is cheap to compute.

**Decision:** The bulk ingestion path bypasses embedding computation. Store a zero vector or NULL embedding as a placeholder. The `search_vector` (tsvector) is still computed. The `idempotency_key` dedup works independently of embeddings.

**Implementation:** The bulk endpoint calls a lightweight version of `_store_fact` (or passes an `embed=False` flag) that skips the `embedding_engine.embed()` call. If the memory module later adds batch-embed capabilities, a backfill job can retroactively compute embeddings.

**Alternative rejected — batch embed in one forward pass:** SentenceTransformer supports `encode(batch)`, but this requires changes to the shared embedding engine and the `store_fact` interface. Higher lift, and the benefit is marginal since transactions don't need semantic search.

**Alternative rejected — async processing:** Accept batch, return job ID, poll for completion. Adds significant complexity (job queue, status endpoint) for a feature that's fine synchronous without embeddings.

### 7. Request/response schema for bulk endpoint

**Request body:**
```json
{
  "account_id": "optional-account-uuid",
  "source": "csv-import",
  "transactions": [
    {
      "posted_at": "2025-01-15T00:00:00Z",
      "merchant": "Whole Foods",
      "amount": "-47.32",
      "currency": "USD",
      "category": "groceries",
      "description": "optional note",
      "payment_method": "optional"
    }
  ]
}
```

**Response body:**
```json
{
  "total": 100,
  "imported": 95,
  "skipped": 4,
  "errors": 1,
  "error_details": [
    {"index": 42, "reason": "invalid date format"}
  ]
}
```

Amount sign convention: negative = debit, positive = credit (same as `record_transaction_fact`).

## Risks / Trade-offs

**[Risk] LLM generates broken script on first attempt** → Mitigation: The skill instructs a verify-then-fix loop. The LLM reads stderr, patches the script, and reruns. Most CSV parsing errors are detectable from Python tracebacks.

**[Risk] Genuine duplicate transactions deduped as false positives** → Mitigation: Report skipped count prominently. Skill instructs the LLM to flag high skip rates. Future enhancement: add an `allow_duplicates` flag.

**[Risk] Large CSV (>50K rows) causes slow import** → Mitigation: Batch size of 100 keeps individual requests fast. The endpoint processes batches independently. No timeout risk per batch.

**[Risk] Script uses non-stdlib imports that aren't available** → Mitigation: Skill explicitly constrains to stdlib-only. If the LLM still uses `requests`, the error is self-correcting (ImportError → LLM rewrites with `urllib`).

**[Risk] Dashboard API not running when script executes** → Mitigation: Script should verify connectivity (HEAD request) before processing rows. Skill instructs this check.

**[Trade-off] No merchant normalization** → Accepted. CSV merchant names are messy but at least consistent within a single statement. Normalization is a separate feature with its own complexity (fuzzy matching, category inference). Raw merchant names are stored as-is.

**[Risk] Partial batch failure creates confusing re-run** → If the script fails at batch 25/50, re-running dedupes the first 2,400 rows as "skipped" while importing the remaining 2,600. The summary looks like a dedup problem. Mitigation: the response distinguishes skip reasons (`"duplicate"` vs `"invalid_date"`), and the skill instructs the LLM to explain re-run semantics to the user.

**[Trade-off] Cross-source dedup depends on normalization** → Email-sourced transactions have clean merchant names ("Whole Foods") while CSV-sourced have raw names ("WHOLEFDS MKT #10456"). Fuzzy substring matching between these is unreliable. Cross-source dedup works best AFTER merchant normalization has run, when both sources use the same canonical names. The cross-source dedup bead should document this sequencing. For initial import of historical data where no email-sourced transactions exist, this is a non-issue.
