# v1 Scope

This document defines what v1 ships and what it defers. If a capability is not
listed under "What v1 Ships," it is not in v1. Scope debates end here.

## What v1 Ships

### Core Infrastructure

- **Multi-butler daemon** --- all domain butlers running as persistent async
  daemons in a single deployment.
- **Switchboard routing** --- LLM-based message classification and dispatch to
  domain butlers, with thread affinity and fanout support.
- **Module system** --- pluggable capability units with tool registration,
  migrations, lifecycle hooks, and topological dependency resolution.
- **Task scheduler** --- cron-driven scheduling with deterministic and
  LLM-dispatched modes.
- **LLM CLI spawner** --- ephemeral session management with locked-down MCP
  configs, concurrency limits, and model routing.
- **Session logging** --- append-only record of every invocation with trigger
  source, tool calls, token usage, and duration.
- **State store** --- per-butler key-value store backed by PostgreSQL JSONB.

### Butlers

- **Switchboard** --- ingress routing and message classification.
- **General** --- freeform data storage, collections, catch-all assistance.
- **Health** --- measurements, medications, conditions, symptoms, nutrition,
  research.
- **Relationship** --- contact management, interaction tracking, life events,
  relationship context.
- **Finance** --- financial signal tracking and awareness.
- **Education** --- learning tracking and knowledge management.
- **Travel** --- trip planning and travel context.
- **Home** --- home automation awareness and environmental context.
- **Messenger** --- outbound notification delivery.

### Modules

- **Memory** --- tiered storage (Eden, Mid-Term, Long-Term) with entity
  resolution, LRU-based promotion/eviction, and cross-butler fact sharing.
- **Calendar** --- Google Calendar integration for event management.
- **Contacts** --- shared identity registry with cross-channel resolution.
- **Email** --- IMAP/SMTP integration for email-aware butlers.
- **Telegram** --- Telegram bot integration for interactive messaging.
- **Approvals** --- safety gates for sensitive tool calls requiring user
  confirmation.
- **Pipeline** --- multi-step processing workflows.
- **Metrics** --- operational metrics collection.
- **Mailbox** --- inter-butler message delivery.
- **Self-healing** --- crash recovery and automatic restart.

### Connectors

- **Telegram bot** --- long-polling adapter for Telegram Bot API.
- **Telegram user client** --- Telethon-based adapter for user account messages.
- **Gmail** --- push notification or periodic IMAP adapter for Gmail.
- **Discord** --- websocket adapter for Discord events.
- **Heartbeat** --- periodic health check connector.
- **Live listener** --- real-time audio transcription connector.

### Dashboard

- **FastAPI backend** (port 41200) with per-butler API routes, session browser,
  contact management, OAuth credential configuration, and module settings.
- **Vite frontend** (port 41173 in development) with real-time butler status,
  session logs, contact and identity views, and ingestion monitoring.

### Identity System

- **Shared contacts registry** --- canonical contact table with roles and entity
  linkage.
- **Cross-channel identity resolution** --- mapping Telegram chat IDs, email
  addresses, and Discord IDs to canonical contacts.
- **Owner bootstrapping** --- automatic owner contact creation on first startup.

### Observability

- **OpenTelemetry instrumentation** --- traces and metrics from daemon, spawner,
  and modules.
- **Grafana Alloy** --- telemetry collection and routing.
- **Tempo** --- distributed trace storage and querying.
- **Prometheus** --- metrics scraping and alerting.

## What v1 Defers

These are explicitly out of scope. They may become v2 goals, but no v1 work
should be designed to "prepare for" them at the cost of v1 simplicity.

- **Multi-user support.** No user accounts, no tenant isolation, no shared
  instances. One user, one deployment.
- **Mobile application.** The dashboard is web-only. No native iOS or Android
  client.
- **Voice interface.** No speech-to-text or text-to-speech integration at the
  butler level (the live-listener connector is transport, not a voice UI).
- **Self-hosting installer.** No one-click deployment script, no Helm chart, no
  managed setup wizard. The user sets up the stack manually.
- **Plugin marketplace.** No third-party module distribution, no module registry,
  no plugin API versioning.
- **Offline operation.** LLM API access is required for intelligent operation.
  The daemon persists state locally, but reasoning requires connectivity.
- **End-to-end encryption.** The user owns the database. See
  [security.md](security.md) for the trust model rationale.
- **Multi-LLM orchestration.** Sessions use a single LLM provider per
  invocation. No chain-of-models or ensemble patterns.

## Platform Targets

- **OS:** Linux (primary), macOS (development). Windows is not tested.
- **Runtime:** Python 3.12+
- **Database:** PostgreSQL 15+ with JSONB support
- **Container:** Docker and Docker Compose for deployment
- **LLM runtimes:** Claude Code (primary), with model routing support for
  Codex and Gemini CLI

## Success Criteria

v1 is complete when:

1. All nine butlers run concurrently in a single deployment and handle their
   declared domains.
2. The Switchboard correctly routes messages from Telegram, Gmail, and Discord
   to the appropriate butler with >90% classification accuracy.
3. Scheduled tasks fire reliably on cron cadences with automatic retry on
   transient failures.
4. The memory subsystem stores, retrieves, and consolidates facts across butler
   sessions.
5. The dashboard provides real-time visibility into butler status, sessions,
   contacts, and ingestion flow.
6. The system runs for 7 consecutive days without manual intervention beyond
   LLM API key rotation.
7. The owner uses the system daily for at least three domains (e.g., health
   tracking, relationship context, general assistance).

## Anti-Patterns

- Building v1 features with multi-user "hooks" that add complexity now.
- Deferring reliability work because "it is just v1."
- Expanding connector coverage before existing connectors are solid.
- Adding modules that no butler in the roster actually uses.
- Designing abstractions for plugin extensibility before the internal module
  API is stable.
