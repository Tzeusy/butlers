# v1 Scope

This document defines what v1 ships and what it defers. If a capability is not
listed under "What v1 Ships," it is not in v1. Scope debates end here.

## What v1 Ships

### Core Infrastructure

- **Multi-butler daemon** --- all domain butlers running as persistent async
  daemons in a single deployment.
- **Switchboard routing** --- LLM-based message classification and dispatch to
  domain butlers, with thread affinity and fanout support.
- **Module system** --- pluggable capability units with tool registration,
  migrations, lifecycle hooks, and topological dependency resolution.
- **Task scheduler** --- cron-driven scheduling with deterministic and
  LLM-dispatched modes.
- **LLM CLI spawner** --- ephemeral session management with locked-down MCP
  configs, concurrency limits, and model routing.
- **Session logging** --- append-only record of every invocation with trigger
  source, tool calls, token usage, and duration.
- **State store** --- per-butler key-value store backed by PostgreSQL JSONB.

### Staffers

Infrastructure agents that serve the ecosystem rather than user life domains.
Declared with `type = "staffer"` in `butler.toml`. Excluded from user-message
routing and briefing contribution.

- **Switchboard** --- ingress routing and message classification.
- **Messenger** --- outbound notification delivery.
- **QA** --- automated error detection, investigation dispatch, and
  anonymized fix pipeline.

### Butlers

Domain specialists that serve the user directly. Each has a manifesto defining
its scope and personality.

- **General** --- freeform data storage, collections, catch-all assistance.
- **Health** --- measurements, medications, conditions, symptoms, nutrition,
  research.
- **Relationship** --- contact management, interaction tracking, life events,
  relationship context, and shared identity registry with cross-channel
  resolution.
- **Finance** --- financial signal tracking and awareness.
- **Education** --- learning tracking and knowledge management.
- **Travel** --- trip planning and travel context.
- **Home** --- home automation awareness and environmental context.
- **Lifestyle** --- taste, music, entertainment, food preferences, hobbies, and daily routines.
- **Chronicler** --- retrospective reconstruction of lived past time from other
  butlers' timestamped evidence (sessions, completed calendar instances, durable
  Spotify summaries, etc.). Projects point events and overlapping episodes with
  preserved source provenance, precision, and correction history. Does not plan,
  ingest externally, or notify; routes only explicit retrospective requests
  (see RFC 0014). The operational `/api/timeline` route (cross-butler session
  and notification stream) remains distinct; Chronicler adds a new
  `/api/chronicler/*` namespace for reconstructed time.

### Modules

- **Memory** --- tiered storage (Eden, Mid-Term, Long-Term) with entity
  resolution, LRU-based promotion/eviction, and cross-butler fact sharing.
- **Contacts** --- tools over the shared contacts registry for managing
  people, their per-channel identifiers (Telegram, email, Discord), and
  contact roles, powering the Identity System.
- **Calendar** --- Google Calendar integration for event management.
- **Email** --- IMAP/SMTP integration for email-aware butlers.
- **Telegram** --- Telegram bot integration for interactive messaging.
- **Approvals** --- safety gates for sensitive tool calls requiring user
  confirmation.
- **Pipeline** --- multi-step processing workflows.
- **Metrics** --- operational metrics collection.
- **Mailbox** --- inter-butler message delivery.
- **Self-healing** --- crash recovery and automatic restart.
- **Home Assistant** --- bidirectional smart home control via WebSocket/REST,
  entity state cache, area/registry awareness, command audit logging.
- **Google Drive** --- file read/write/search/organize via Google Drive API,
  auto-created `butlers/` folder hierarchy for butler output.
- **Google Health** --- wellness data access (sleep, HR, HRV, SpO2, steps,
  activity, VO2 max) exposed as MCP tools, backed by the Google Health
  ingestion connector.
- **WhatsApp** --- output-only MCP tools for sending/replying via Go sidecar.
- **Steam**: Steam account integration exposing game library, playtime,
  recently-played, and online-status awareness via the Steam Web API.
- **Insight broker** --- Switchboard-side module for proactive insight delivery:
  candidate submission, dedup, budget gating, adaptive delivery, anti-spam.

### Connectors

- **Telegram bot** --- long-polling adapter for Telegram Bot API.
- **Telegram user client** --- Telethon-based adapter for user account messages.
- **Gmail** --- push notification or periodic IMAP adapter for Gmail.
- **Discord** --- websocket adapter for Discord events.
- **Heartbeat** --- periodic health check connector.
- **Live listener** --- real-time audio transcription connector.
- **Google Calendar** --- sync-token polling adapter for calendar change events.
- **Google Drive** --- changes-list polling adapter for file metadata events.
- **Home Assistant** --- WebSocket/REST adapter for smart home events with
  domain and significance filtering.
- **OwnTracks** --- HTTP webhook receiver for location and waypoint events.
- **Spotify** --- playback state polling adapter for listening events and
  session aggregation.
- **Google Health** --- polling adapter for wellness data (sleep, HR, HRV, SpO2,
  steps, activity, VO2 max) via the Google Health API, reusing the existing
  Google OAuth credential pipeline.
- **WhatsApp user client** --- passive inbox ingestion via whatsmeow Go sidecar.
- **Steam**: polling adapter for the Steam Web API (recently-played, online
  status, friends, game library) with per-data-type cadence.

### Dashboard

- **FastAPI backend** (port 41200) with per-butler API routes, session browser,
  contact management, OAuth credential configuration, and module settings.
- **Vite frontend** (port 41173 in development) with real-time butler status,
  session logs, contact and identity views, and ingestion monitoring.
- **Settings Console** (`/settings`) — Dispatch-language panel grid with
  attention strip; overview of system health, spend, approvals, and model
  status. Implemented by the Settings Dispatch Console refactor
  (openspec/changes/redesign-settings-dispatch-console/).
  - `/settings/models` — model catalog, tier-grouped with per-row priority
    controls, enable toggles, and verify-all tooling.
  - `/settings/spend` — spend dashboard with MTD forecast, butler/model
    breakdown, routing rules, and ceiling configuration.
  - `/settings/permissions` — permissions matrix, audit reel, data-ops
    (export and wipe), and webhook registry.
- **Dashboard audit log** — append-only `public.audit_log` table recording
  every configuration mutation (actor, action, target, reason, IP,
  request\_id). No deletes, indefinite retention. Exposed at
  `GET /api/audit-log`. All settings mutations call `audit.append()`.
- **Webhooks** — outbound HMAC-SHA256-signed POST delivery with per-endpoint
  retry policy, test endpoint, and failure tracking.
- **Data ops** — encrypted export (scoped zip, 60-minute signed URL) and
  owner-confirmed wipe (`WIPE EVERYTHING IRREVERSIBLY` phrase required).

### Identity System

- **Shared contacts registry** --- canonical entity registry with contact
  predicates.
- **Cross-channel identity resolution** --- mapping Telegram chat IDs, email
  addresses, and Discord IDs to canonical entities via contact predicates
  (`entity_facts`).
- **Owner bootstrapping** --- automatic owner entity creation on first startup.

### Situational Awareness

- **Context bus** --- shared `user_context` table with TTL-bounded signals
  (traveling, sleeping, meeting, focused, etc.) readable by all butlers.
  Pull-based, opt-in context checking at decision points (RFC 0009).
- **Proactive insight delivery** --- three-phase pipeline: butler-side
  generation, Switchboard-side brokering (dedup, budget, adaptive delivery),
  and digest delivery via Messenger. Global daily budget, per-key cooldowns,
  one-way adaptive ratchet, auto-off on sustained disengagement (RFC 0011).

### Observability

- **OpenTelemetry instrumentation** --- traces and metrics from daemon, spawner,
  and modules.
- **Grafana Alloy** --- telemetry collection and routing.
- **Tempo** --- distributed trace storage and querying.
- **Prometheus** --- metrics scraping and alerting.

## What v1 Defers

These are explicitly out of scope. They may become v2 goals, but no v1 work
should be designed to "prepare for" them at the cost of v1 simplicity.

- **Multi-user support.** No user accounts, no tenant isolation, no shared
  instances. One user, one deployment.
- **Mobile application.** The dashboard is web-only. No native iOS or Android
  client.
- **Voice interface.** No speech-to-text or text-to-speech integration at the
  butler level (the live-listener connector is transport, not a voice UI).
- **Self-hosting installer.** No one-click deployment script, no Helm chart, no
  managed setup wizard. The user sets up the stack manually.
- **Plugin marketplace.** No third-party module distribution, no module registry,
  no plugin API versioning.
- **Offline operation.** LLM API access is required for intelligent operation.
  The daemon persists state locally, but reasoning requires connectivity.
- **End-to-end encryption.** The user owns the database. See
  [security.md](security.md) for the trust model rationale.
- **Multi-LLM orchestration.** Sessions use a single LLM provider per
  invocation. No chain-of-models or ensemble patterns.

## Platform Targets

- **OS:** Linux (primary), macOS (development). Windows is not tested.
- **Runtime:** Python 3.12+
- **Database:** PostgreSQL 15+ with JSONB support
- **Container:** Docker and Docker Compose for deployment
- **LLM runtimes:** Claude Code (primary), with model routing support for
  Codex and Gemini CLI

## Success Criteria

v1 is complete when:

1. All three staffers and nine domain butlers (including Chronicler) run
   concurrently in a single deployment and handle their declared
   responsibilities.
2. The Switchboard staffer correctly routes messages from all active connectors
   (Telegram, Gmail, Discord, Google Calendar, Google Drive, Home Assistant,
   OwnTracks, Spotify, WhatsApp) to the appropriate domain butler with >90%
   classification accuracy. Lifestyle-only ingestion connectors (e.g. Steam)
   are intentionally excluded from this routing-accuracy set, since they feed
   ambient awareness rather than serve as routing targets.
3. Scheduled tasks fire reliably on cron cadences with automatic retry on
   transient failures.
4. The memory subsystem stores, retrieves, and consolidates facts across butler
   sessions.
5. The dashboard provides real-time visibility into butler status, sessions,
   contacts, and ingestion flow.
6. The system runs for 7 consecutive days without manual intervention beyond
   LLM API key rotation.
7. The owner uses the system daily for at least three domains (e.g., health
   tracking, relationship context, general assistance).
8. Proactive insights are delivered at a sustainable cadence without manual
   tuning, and the adaptive ratchet correctly reduces frequency on
   disengagement.

## Anti-Patterns

- Building v1 features with multi-user "hooks" that add complexity now.
- Deferring reliability work because "it is just v1."
- Expanding connector coverage before existing connectors are solid.
- Adding modules that no butler in the roster actually uses.
- Designing abstractions for plugin extensibility before the internal module
  API is stable.
